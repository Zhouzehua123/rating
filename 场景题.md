# 1.你这个系统线上最容易出事故的点是什么？你怎么提前防？

其实我觉得我们这个系统线上最容易出问题的地方，主要还是在Redis这一层。

因为Redis在我们架构里是做查询加速的，绝大部分商家评价的查询都会优先走Redis。

如果Redis挂了，虽然不会影响到业务数据的写入，也不会丢数据，但所有的读请求就会直接打到ES上，

这时候ES的压力会瞬间增大，极端情况下甚至可能把ES打挂。

所以针对这个风险，我做了几方面的防护。首先，Redis本身我们是可以用集群+哨兵或者主从的高可用部署，

单点故障可以自动切换，发现异常也能自动重启，最大程度保证Redis的可用性。

但即使Redis真的挂了，我们还有兜底方案。我们的查询链路是三层的，

先查Redis，Redis没命中就查ES，如果ES也有问题，还可以降级查MySQL，

保证数据一定能查到，虽然体验会慢一些，但不会影响核心业务。

另外，为了防止缓存击穿和雪崩，我们对不同热度的商品设置了不同的缓存策略。

比如热门商品的缓存时间会设置得很长，甚至可以异步刷新、永不过期；

一般商品10分钟，冷门商品1分钟。所有缓存key的过期时间还会加上一个随机偏移，避免同一时刻大面积失效。

还有一个很关键的点，就是我们用了Go的singleflight组件。

这个东西可以把同一个key的并发回源合并成一次，其他请求都等着结果返回，

这样即使热点key突然失效，也不会有大量请求同时打到ES，极大降低了回源压力。

最后，Redis恢复以后，我们会自动预热热点数据，比如热门商家的评价列表，快速把缓存补回来，缩短恢复窗口。

（整体来说，我们的目标就是：即使Redis出问题，系统依然可用，数据不丢，体验只是略慢，通过多层兜底和热点治理，保护ES和MySQL不被打挂，保障核心业务稳定。）

# 2.用户说刚评价的消息看不到，链路有延迟怎么办？

如果用户说“刚发评论看不到”，其实这个问题在我系统里是有分场景处理的。

对于用户自己刚发的评论，我们是直接走MySQL查的，也就是用userID查自己的评价列表，

这样可以保证用户写完评论后，基本是立刻可见的，体验上是强一致的。

但对于商家侧的评价列表，或者其他用户的查询，我们是走Redis+ES这条高性能链路，

数据是通过Kafka异步同步过去的，所以会有一个短暂的延迟。这个延迟一般在秒级，

极端情况下可能会到几十秒，但我们会监控延迟窗口，保证大部分场景下用户感知不到。

其实这个思路和支付场景很像，关键链路必须强一致，比如用户支付结果、评论刚写完的可见性，

其他高并发的查询就可以用异步扩散、最终一致来换取性能。

# 3.singleflight 解决了什么？它解决不了什么？

singleflight 主要解决的是高并发场景下，同一个 key 的缓存击穿问题。

比如某个热点评价的缓存刚好过期了，突然有成百上千个请求同时进来，

如果没有控制，这些请求会一起回源查 ES 或 MySQL，瞬间把后端打挂。

用了 singleflight 之后，同一时刻只会放行一个请求去查后端，其他请求都等着，

等第一个请求查完把结果写回缓存，大家一起复用这个结果，这样就能大幅降低回源压力，保护后端服务。

但 singleflight 也有它的局限。首先最大的特点就是，它只能在单机进程内合并请求，

不能跨多台机器全局合并。如果热点 key 在多台服务实例上同时失效，还是会有多个回源请求。

其次，singleflight 只能解决同 key 的并发回源，

不能防止缓存雪崩、热点切换、慢查询等问题。所以它只是缓存击穿防护的一个环节，

还需要配合限流、降级、异步刷新等手段一起用，才能真正保障系统稳定。


解决方法：

分布式锁就是很常见的一种做法，热点 key 回源前，大家先去抢锁，

只有抢到锁的那一个实例才去查后端，其他实例就等着，等锁释放后直接读缓存。

这样就把 singleflight 的合并能力扩展到了多机多进程，真正做到了全局只回源一次。

其实分布式锁和 singleflight 各有适用场景，不能一刀切。
singleflight 的优点是极快，基本没有额外开销，非常适合在单机、单进程内做高并发合并，

比如我们服务实例本身的并发请求。但它的局限就是只能在本机生效，跨实例就无能为力了。

分布式锁虽然能做到全局合并，但它的开销会大一些，比如抢锁、等待锁释放，

网络延迟也会拉长整体回源时间。如果热点 key 很多、回源很频繁，

分布式锁反而可能成为瓶颈，甚至影响正常请求。

所以一般来说， **大部分场景下只用 singleflight 就够了** ，

比如热点 key 不多、服务实例数量有限、回源压力可控的时候，这样既快又省资源。

但如果遇到极端高并发、热点 key 特别集中、服务实例数量非常多，

或者业务对后端保护要求极高，这时候可以考虑把分布式锁和 singleflight 组合起来用。

比如本机先用 singleflight 合并，跨实例再用分布式锁兜底，这样既保证了效率，也能防止极端情况下的全局击穿。

# 4.评价“重复提交”怎么做幂等？

（比如用户网络不好或者前端接口超时，可能会点很多次发送后端）

我这里是在不同层做了幂等保护：

首先是业务唯一键校验。比如每个用户对同一个订单只能评价一次，我们会用 userID + orderID 作为唯一索引，收到新请求时先查一下有没有已经存在的评价，如果有就直接返回，不再重复写入。

然后是数据库层面的唯一约束。我们会在评价表里加一个唯一索引（userID, orderID），即使并发下有两个请求同时进来，也只有一个能成功插入，另一个会被数据库拦截。

如果业务有更高的幂等要求，还可以让前端或者调用方传一个幂等 token，每次请求都带上，后端用 token 做一次性校验，防止更复杂的重复提交场景。这样多层保护下来，无论是前端误操作、网络抖动还是并发写入，都能保证评价不会被重复创建，就实现了幂等性。

# 5.怎么做的压测，性能瓶颈在哪？

我在做性能压测的时候，最早其实是直接用MySQL主从架构做读写分离。

刚开始没觉得有问题，但后来发现就是用户和商家都可以按关键词搜索评价内容，比如搜“质量”这种词。

因为SQL里做这种模糊查询只能用 `LIKE '%关键词%'`，我查了很多资料，

发现这种写法其实会导致索引失效，变成全表扫描，性能很容易出问题。

为了验证这个猜想，我专门写了个Go脚本，批量插入了10万条评价数据，

模拟了多个商家和用户的真实场景。然后我开了MySQL的慢查询日志，

设置了0.5秒的阈值，做了三种查询实验：精准查、前缀模糊查和全模糊查。

结果发现全模糊查询果然会全表扫描，响应时间基本都在600毫秒到1秒多。

而且商家侧的查询还涉及到评价、回复、申诉三张表的关联，压力更大。

我用JMeter做了简单压测，100个并发用户，每人10次查询，平均响应时间1.2秒，

CPU占用也很高，慢查询日志里全是超时的SQL。

所以我就决定优化架构，借鉴了社区的CQRS思路，把读写彻底分开。

写操作还是走MySQL，保证事务一致性。读操作的话，热数据直接走Redis缓存，

冷数据和复杂查询就走Elasticsearch，利用ES的倒排索引做全文检索。

数据同步是通过Canal监听MySQL binlog，写到Kafka，再由review-job服务消费写入ES。

优化后我又做了一轮压测，发现ES的响应时间基本都在100毫秒以内，

CPU压力也小了很多，整体性能提升非常明显。

所以总结下来，性能瓶颈主要在于MySQL做全模糊查询和多表关联，

优化后通过Redis和ES分担了大部分读压力，系统的高并发能力提升了好几个量级。

# 6.线上有一波流量突然暴涨，会出现什么问题？要怎么办？

如果线上突然遇到一波流量暴涨，其实最容易出问题的还是在缓存、ES和数据库这三层。

首先，Redis 作为缓存层，如果热点 key 失效或者 Redis 本身挂掉，所有的流量就会瞬间回源到后端，

容易出现缓存击穿或者雪崩，导致 ES 和 MySQL 压力暴增。

接下来就是 ES 检索层。如果回源流量太大，ES 承压过重，可能会出现查询延迟变高，

甚至服务不可用，直接影响商家和用户的查询体验。

如果 ES 也顶不住，流量就会继续回源到 MySQL，这时候主库的压力会非常大，

极端情况下甚至会影响到写入和整体业务的稳定性。

针对这种情况，我一般会分几步应对。第一时间肯定是做限流和降级，保护核心链路，

非核心的请求可以直接返回兜底数据或者友好提示，防止雪崩扩散。

然后，Redis 层会有热点 key 的异步刷新和预热机制，尽量避免大面积失效。

ES 层会设置 QPS 和延迟的阈值，触发自动降级，必要时只保留核心查询。

MySQL 层会有只读实例兜底，同时监控慢查询和连接数，防止主库被打挂。

# 7.这个项目里你做的最关键决策是什么？如果重来一次会怎么改？

我觉得在这个项目里，我做的最关键的决策，其实就是把读写彻底分离，

采用了 MySQL 保证强一致写入、ES 做高性能检索、Redis 做热点加速的三层架构。

这个决策的核心出发点，是希望既能保证数据的可靠性和一致性，又能支撑高并发下的查询性能，满足业务增长的需求。

如果让我重来一次，我会在几个地方做优化。首先，

状态流转和业务枚举会全部抽象成统一的状态机和常量，避免后期维护时出现魔法数字和状态混乱。

其次，配置和敏感信息会全部用环境变量和密钥管理，提升安全性。

还有就是观测性方面，早期就会把埋点、监控、告警体系搭建好，方便后续快速定位和复盘问题。

整体来说，这个架构决策让系统具备了弹性和扩展性，

但如果有机会重来，我会在规范性和可维护性上做得更好。

# 8.如果让你带 2 个同学把它产品化，你的阶段性计划是什么？

一开始，前30天我会带大家把系统的稳定性和规范性打牢，

比如把所有的状态码、配置、日志、监控都梳理清楚，保证出问题能第一时间发现和定位。

接下来30天，我们会重点做功能完善和体验优化，

比如补齐接口文档、完善异常处理、把常用的查询和操作做成后台页面，方便业务方直接用。

最后30天，我们会关注扩展性和成本优化，比如做多租户支持、权限体系、自动化运维脚本，甚至考虑云上部署和弹性扩容。

整个过程中，我会让大家轮流负责不同模块，既能锻炼能力，

也能保证每个人都能独立hold住一块业务。最终目标就是让这个系统不仅能跑，还能真正支撑业务上线和持续迭代。
